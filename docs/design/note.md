# Note機能

## Context

CC Pocketのチャット欄のmarkdown機能を活かした軽量メモ機能。
エージェント不在のセッションとして、既存のチャットUIをそのまま再利用する。

考えが固まったら、Noteの内容をコンテキストとしてClaude Code / Codexセッションに変換できる。

### Prompt Draftとの違い

Prompt Draft（`docs/design/prompt-draft.md`）はリッチエディタによるプロンプト推敲ツール。
実装に進めなかった主な理由:

- **実装コストが大きい** — エディタ・一覧・画像管理すべて新規（10ファイル+）
- **導線がCC Pocketのシンプルさを損なう** — AppBarにアイコン追加→独立画面→エディタと、本流とは別の導線が必要だった

NoteはチャットUIをそのまま使い、思考の流れを時系列で記録する。
New Sessionの選択肢に1つ増えるだけで、作成・閲覧・一覧すべてが既存の流れに統合される。

## ユースケース

1. **実装前の思考整理** — バグの原因仮説や設計案をチャット形式で書き出し、まとまったらセッション開始
2. **要件・仕様メモ** — MTGで聞いた要件をさっと投稿、後でエージェントに渡す
3. **プロンプトの下書き** — 複雑な指示を段階的に組み立てて、完成したらセッションに変換
4. **調査ログ** — 手動調査の結果を記録しながら、エージェントに任せる部分を切り出す
5. **振り返り** — セッション後の学びや気づきを書き留める

## 設計判断

| 判断 | 決定 | 理由 |
|------|------|------|
| UI | 既存チャットUIを流用 | 新規エディタ不要、markdown表示がそのまま使える |
| メッセージ編集 | **やらない** | チャットUIに編集機能を入れると複雑化。思考の流れをそのまま残す価値がある |
| メッセージ削除 | **やる** | 変換前にノイズを除去できる。長押し→削除 |
| メッセージコピー | **やる** | 編集の代替。コピー→修正して再投稿は自然な流れ |
| 画像添付 | **やる（上限付き）** | 1メッセージあたり上限を設ける（Claude APIの20枚制限に合わせる）。バリデーションは投稿時のみ（シンプルさ優先）。変換時の合計チェックはしない（APIエラーに委ねる） |
| タグ・フォルダ | **やらない** | セッション一覧のフィルタで十分 |
| 検索 | **やらない（初期）** | 既存セッション検索で十分 |
| リッチテキストエディタ | **やらない** | markdownをプレーンテキストで書く、既存のまま |
| タイトル | **やる** | 一覧での識別用。変換時にセッション名として引き継ぐ |
| プロジェクト紐づけ | **やる** | 変換時にプロジェクトパスをそのまま引き継ぐ |
| 永続化 | SQLite（sqflite） | プロンプト履歴で使用実績あり。AIセッションとは別テーブル |
| セッション変換後のNote | **残す（手動削除）** | 変換はコンテキストのコピー。元Noteは振り返り用に残す |
| オフライン利用 | **初期はやらない** | Bridge未接続時にセッション一覧を表示する分岐が必要になるため後回し。トークンlimit到達時に使えるだけでも価値がある |

## セッション一覧のフィルタ

既存の「All AI tools」フィルタをリネーム:

| 現在 | 変更後 |
|------|--------|
| All AI tools | **All Types** |
| Claude Code | Claude Code |
| Codex | Codex |
| （なし） | **Note** |

## セッション変換フロー

1. Note画面で「セッションに変換」アクション
2. 変換先を選択（Claude Code / Codex）
3. Noteのタイトル → セッション名、プロジェクト → プロジェクトパスとして引き継ぎ
4. 全メッセージをコンテキストとして新規セッションに渡す
5. 元Noteはそのまま残る

### コンテキスト形式

| 優先度 | 形式 | 条件 |
|--------|------|------|
| 理想 | **会話履歴風** — 各メッセージをuser messageとして時系列に並べる | SDKが履歴/会話コンテキストの注入をサポートしている場合 |
| フォールバック | **1テキストまとめ** — markdown形式で全メッセージを結合し、最初のuser messageとして送る | SDK非対応の場合 |

フォールバック時のフォーマット例:

```
## Note: {タイトル}

### 2024-01-15 10:30
{メッセージ1}

### 2024-01-15 10:35
{メッセージ2}
```

### 変換時に渡す構成要素

| 要素 | 変換時の扱い |
|------|------------|
| テキスト（全メッセージ） | コンテキストとして渡す |
| 画像 | テキストと一緒に渡す |
| プロジェクトパス | セッションの `projectPath` に引き継ぎ |
| タイトル | セッション名に引き継ぎ（未サポートのツールでは無視） |

## データモデル

AIセッションとNoteはDBテーブルを完全に分離し、一覧表示時に抽象モデルでマージする。

```
// 一覧表示用の抽象モデル
sealed class SessionListItem {
  String get id;
  String get title;
  DateTime get updatedAt;
  String? get projectPath;
}

// AIセッション（既存・Bridge経由で取得）
class AiSessionItem extends SessionListItem { ... }

// Note（新規・ローカルSQLiteから取得）
class NoteItem extends SessionListItem { ... }
```

- **AIセッション**: 既存のBridge `list_sessions` で取得
- **Note**: ローカルSQLiteから取得
- **一覧表示**: 両ソースを `updatedAt` でマージソートして統合
- 統合ロジックは Provider/Repository 層に配置

## 未決事項

- 一覧統合ロジックの具体的な配置場所（既存の `session_list` feature 内のどこに挟むか）
